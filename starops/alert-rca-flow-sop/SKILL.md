# 通过 STAROps 完成告警 RCA 全链路分析 — SOP Skill

> 本仓 **SOP Skill** 入口：把 [人读版实践文档](https://sls.aliyun.com/doc/starops/practices/alert-rca-flow/article.html) 的 SOP 固化为 Agent 可加载形态。
>
> 区分：本实践无 meta skill 样品（产物是 RCA 报告，不是可执行业务 Skill）。

## SOP 概览

6 phase 端到端流程，章节顺序与 [人读版实践文档 步骤一～步骤六](https://sls.aliyun.com/doc/starops/practices/alert-rca-flow/article.html) 一一对应：

1. 告警分诊 — 判断告警是否值得查
2. 数据确认 — 排除误报，确认告警真实性
3. RCA 根因定位 — 定位到底是什么坏了
4. 证据采集 — 收集支撑证据链
5. 修复建议 — 止血 + 根治方案（L0 只出建议）
6. 复盘治理 — 防止再发生的改进措施

## 步骤 1：告警分诊

**Agent 指令**：

1. 获取告警规则详情（指标 + 阈值 + 持续时间 + 多级阈值分级）
2. 查询告警事件（触发时间 + 当前值 + 持续状态）
3. 查询告警主体的实体详情（服务 / 接口 / Pod）
4. 查询错误 trace 并诊断（定位根因服务 + 错误原因）
5. 综合输出 4 段分诊报告：触发规则 / 影响范围 / 严重程度（P0-P3 + 4 维度依据）/ 响应建议

**用户提问示例**：

```
@告警-<告警名称或 ID> 帮我做告警分诊：
  1. 这条告警的触发规则是什么（指标 + 阈值 + 持续时间）
  2. 影响范围（涉及哪些服务 / 实例 / 用户群）
  3. 严重程度判断（P0-P3 建议 + 依据）
  4. 是否需要立即响应，还是可以延后观察
```

**输出要求**：

- 触发规则段：列出多级阈值表（INFO/WARNING/ERROR/CRITICAL），标出当前触发级别
- 影响范围段：从 trace 还原调用链路，列出告警主体 + 根因服务 + 影响接口 + 错误规模
- 严重程度段：P0-P3 建议 + 4 维度依据（错误率 / 影响面 / 趋势 / 根因明确度）
- 响应建议段：立即响应 / 延后观察 + 理由 + 具体后续动作

**注意事项**：Phase 1 可包含初步 RCA（AI 可能顺手做 trace 诊断定位根因），不严格切割。

## 步骤 2：数据确认

**Agent 指令**：

1. 查询告警触发时间点附近的原始指标数据（错误率 + 异常数 + 请求量）
2. 查询告警触发前 1 小时的基线数据（同指标集）
3. 对比基线 vs 告警时，计算变化幅度，判断数据真实性（排除采集延迟 / 数据缺失）
4. 评估阈值设置合理性（基线与阈值的关系 + 调整建议）
5. 排查关联告警：同一规则其他事件 / 同一实体其他规则 / 下游服务 / 根因服务
6. 综合输出 3 段报告 + 综合结论

**用户提问示例**：

```
@告警-<告警名称或 ID> 确认这条告警的数据真实性：
  1. 告警触发时的指标数据是否真实（排除采集延迟 / 数据缺失 / 误报）
  2. 告警阈值设置是否合理（对比历史基线）
  3. 是否有其他关联告警（同一根因触发的多条告警）
```

**输出要求**：

- 指标数据真实性段：4 维度验证表（数据连续性 / 完整性 / 采集延迟 / 数值合理性）
- 阈值合理性段：基线对比表（历史基线 vs 告警时 + 变化幅度 + 当前阈值 + 评估）+ 阈值问题分析 + 调整建议
- 关联告警排查段：4 层排查范围 + 结论 + 可能原因
- 综合结论段：三项汇总 + 整体可信度

**注意事项**：阈值偏低问题可作为 Phase 6 复盘治理的输入，在此阶段标注即可。

## 步骤 3：RCA 根因定位

**Agent 指令**：

1. 获取告警主体的调用拓扑（从告警主体出发追踪到根因服务）
2. 查询调用链上每个服务的错误率指标
3. 搜索错误 trace 并诊断 span（定位具体错误类型 + 错误详情）
4. 查看 trace 完整结构（树形，含耗时 + 错误标记）
5. 查询相关服务的异常指标（补充证据）
6. 综合输出 4 段根因报告 + 建议后续动作

**用户提问示例**：

```
@告警-<告警名称或 ID> 定位这条告警的根因：
  1. 从告警主体出发，追踪调用链路到根因服务
  2. 根因服务的具体错误是什么（错误类型 + 错误详情）
  3. 根因的支撑证据（指标异常 + 日志错误 + trace 错误 span）
  4. 给出根因假设（一句话总结）
```

**输出要求**：

- 调用链路追踪段：调用链拓扑图（缩进格式，从告警主体到根因服务）
- 根因服务具体错误段：错误表（服务 + 错误类型 + 错误详情），可能涉及多个服务
- 根因支撑证据段：3 类证据——trace 错误 span（树结构 + 耗时 + 错误标记）+ 指标异常（多服务对比表）+ 关键发现
- 根因假设段：一句话总结（根因服务 + 具体问题 + 触发条件 + 影响路径）
- 建议后续动作：具体修复建议

**注意事项**：
- 根因类型包括系统故障（配置缺失、资源耗尽）和业务逻辑错误（库存不足、参数校验失败）
- Phase 3 可能推翻 Phase 1 的初步 RCA，这是正常现象（Phase 1 是浅层定位，Phase 3 是深入追踪）

## 步骤 4：证据采集

**Agent 指令**：

1. 采集指标证据：查询告警触发时 + 基线对比的完整时间序列
2. 采集日志证据：查询错误日志关键词聚类 + 原文节选（可能在 trace logstore）
3. 采集 Trace 证据：整理完整调用链 + 错误 span 详情（spanName + 耗时 + statusCode）
4. 采集拓扑证据：整理依赖关系图 + 关键依赖状态标注（正常/错误）
5. 生成证据包文档：将 4 类证据整理成 markdown 文件

**用户提问示例**：

```
@告警-<告警名称或 ID> 采集这条告警的完整证据包：
  1. 指标证据（告警触发时 + 基线对比，时间序列截图或数据表）
  2. 日志证据（错误日志原文 + 关键词聚类）
  3. Trace 证据（完整调用链 + 错误 span 详情）
  4. 拓扑证据（告警主体到根因服务的依赖关系图）
```

**输出要求**：

- 指标证据：基线 vs 告警时对比表 + 时间序列关键发现（峰值时间点、变化幅度）
- 日志证据：错误关键词聚类表（关键词 + 出现位置 + 含义）+ 错误日志原文节选
- Trace 证据：完整调用链（缩进格式）+ 错误 span 详情表（spanName + 耗时 + statusCode + 错误详情）
- 拓扑证据：依赖关系图（缩进格式 + 关键依赖状态标注：正常 ✓ / 错误 ⚠）
- 证据包文档：生成 markdown 文件作为完整证据包附件

**注意事项**：
- Phase 4 定位是"整理汇总"，和 Phase 3 的"分析定位"有重叠但视角不同
- 日志可能在 trace logstore（当标准 logstore 无日志时）
- 如果觉得 Phase 4 冗余，可以跳过直接进入 Phase 5

## 步骤 5：修复建议

**Agent 指令**：

1. 回顾 Phase 3 识别的所有根因（主因 + 并发问题）
2. 为每个根因设计短期止血方案（立即执行，降低影响）
3. 为每个根因设计长期根治方案（解决根因，防止复发）
4. 每个方案给出具体操作步骤（SQL / kubectl / 代码片段，不是抽象描述）
5. 每个方案给出风险评估表（风险项 + 等级 + 说明）+ 回滚方案
6. 输出执行优先级表（P0-P3 + 执行时间 + 负责人）
7. 输出验证清单（修复后的验证项 + 预期结果 + 验证方法）

**用户提问示例**：

```
@告警-<告警名称或 ID> 针对这条告警的根因，给出修复建议：
  1. 短期止血方案（立即执行，降低影响）
  2. 长期根治方案（解决根因，防止复发）
  3. 每个方案的具体操作步骤
  4. 风险评估（操作风险 + 回滚方案）
```

**输出要求**：

- 短期止血方案：每个方案含目标 + 具体操作步骤（可执行代码）+ 风险评估表 + 回滚方案
- 长期根治方案：同上结构，覆盖所有已识别根因
- 执行优先级表：P0-P3 + 执行时间 + 负责人
- 验证清单：修复后的验证项 + 预期结果 + 验证方法

**注意事项**：
- L0 只出建议，不执行变更。输出中必须明确声明"以下 SQL/命令/代码仅供参考，需人工确认后执行"
- 修复建议应覆盖 Phase 3 识别的所有根因（主因 + 并发问题），不遗漏

## 步骤 6：复盘治理

**Agent 指令**：

1. 回顾 Phase 1-5 积累的证据（根因、证据包、修复建议）
2. 输出告警规则优化建议（4 维度：阈值调整 / 过滤条件 / 动态基线 / 规则拆分），每个含现状问题 + 优化方案 + 预期收益
3. 输出监控覆盖盲区（4 类：服务 / 指标 / 链路 / 业务），每个含缺失内容 + 风险等级 + 影响
4. 输出预防措施（4 维度：配置管理 / 库存治理 / 错误分类 / 架构韧性），每个含具体措施 + 实施路径
5. 输出 Action Items（10 条任务，含任务描述 + 负责人 + 优先级 P0-P3 + 截止时间 + 验收标准）
6. 输出复盘总结（4 大治理短板 + 4 个量化预期收益）

**用户提问示例**：

```
@告警-<告警名称或 ID> 针对这条告警做复盘治理：
  1. 告警规则优化建议（阈值调整 / 过滤条件 / 动态基线）
  2. 监控覆盖盲区（哪些关联服务/指标没有告警）
  3. 预防措施（防止同类问题再次发生的改进项）
  4. 复盘 Action Items（具体任务 + 负责人 + 截止时间）
```

**输出要求**：

- 告警规则优化段：4 维度优化表（阈值调整 / 过滤条件 / 动态基线 / 规则拆分），每个含现状问题 + 优化方案 + 预期收益
- 监控覆盖盲区段：4 类盲区表（服务 / 指标 / 链路 / 业务），每个含缺失内容 + 风险等级 + 影响
- 预防措施段：4 维度措施表（配置管理 / 库存治理 / 错误分类 / 架构韧性），每个含具体措施 + 实施路径
- Action Items 段：10 条任务表（任务描述 + 负责人 + 优先级 + 截止时间 + 验收标准）
- 复盘总结段：4 大治理短板 + 4 个量化预期收益（告警准确率 / MTTR / 复发率 / 闭环建设）

**注意事项**：
- Phase 6 是闭环的关键，必须基于 Phase 1-5 的真实证据，不能泛泛而谈
- Action Items 必须可执行、可验收、有明确负责人和截止时间
- 复盘总结应含量化预期收益，便于后续跟踪

## 输出与交付

SOP 跑完后客户能拿到：
- **分诊报告**（Phase 1）：触发规则 + 影响范围 + P 级别 + 响应建议
- **数据确认报告**（Phase 2）：数据真实性验证 + 基线对比表 + 关联告警排查 + 综合结论
- **RCA 报告**（Phase 3）：调用链路 + 根因服务错误 + trace/指标/日志证据链 + 根因假设
- **证据包文档**（Phase 4）：4 类证据结构化汇总 markdown 文件
- **修复方案**（Phase 5）：短期止血 + 长期根治方案（含可执行代码）+ 优先级表 + 验证清单
- **复盘报告**（Phase 6）：告警规则优化 + 监控盲区 + 预防措施 + Action Items（含量化预期）

## 失败与回滚

L0 只读实践，无回滚需求。各 Phase 失败处置：

| 失败场景 | 处置 |
|---|---|
| Phase 1 告警实体不存在 | 检查告警名称/ID 是否正确；确认告警规则已接入 |
| Phase 2 基线数据缺失 | 扩大查询时间窗口（如前 24 小时）；确认数据源已接入 |
| Phase 3 trace 数据缺失 | 确认 trace 采集已开启；检查采样率是否过低 |
| Phase 4 标准 logstore 无日志 | AI 会自动转向 trace logstore；如仍无数据，确认日志采集配置 |
| Phase 5 修复建议过于泛泛 | 补充 Phase 3 的证据细节，重新提问 |
| Phase 6 Action Items 缺乏针对性 | 确保 Phase 1-5 在同一 thread 内完成，保留上下文 |

人工介入时机：AI 无法定位根因（Phase 3 返回"未找到"）或修复建议涉及生产变更时，需 SRE 人工接管。

## 召回 Routing

| 用户意图 | 路由到 |
|---|---|
| 告警分诊 / 判断严重程度 | Phase 1 |
| 排除误报 / 验证数据真实性 | Phase 2 |
| 定位根因 / 追踪调用链 | Phase 3 |
| 整理证据 / 生成 RCA 附件 | Phase 4 |
| 修复方案 / 止血措施 | Phase 5 |
| 复盘 / 告警治理 / 预防措施 | Phase 6 |
| 提问方式 / prompt 写法 | 见同仓 `effective-prompts` 实践 |
