# 订单创建失败率多分支诊断

## 使用场景

当用户要求排查订单创建失败率、下单失败、`OrderCreate` 失败、`POST /api/orders` 失败或订单错误码聚集告警时，使用本 Skill。

本 Skill 适合业务交易和错误码相对稳定的场景。根因方向未知、错误码缺失、证据冲突或分支无法覆盖时，升级 InvestigationAgent。

## 前置条件

- 能定位告警时间窗。
- 能绑定订单创建业务交易、订单 API 或 `order-service`。
- ARMS/APM 或 UModel 中能查询应用、接口、上下游依赖和 Trace。
- 失败请求能提取 `error_code`，或能从日志 / Trace / 返回码推断错误族。
- 诊断为 L0 只读，可以执行只读查询和取证，不执行写入 SQL、发布、配置修改、重启、扩缩容等生产变更。

## 执行流程

### 1. 归一告警上下文

先确认：

- 告警名称或告警 ID。
- 触发时间窗。
- 告警指标、阈值、当前值、持续时间、聚合方式。
- 业务交易：优先识别 `OrderCreate`。
- API：优先识别 `POST /api/orders`。
- 应用：优先识别 `order-service`。

缺少时间窗、业务交易、API 或应用时，先要求用户补充信息。

### 2. 查询错误码分布

查询触发窗口和基线窗口：

- 订单创建请求量。
- 订单创建成功率。
- 订单创建失败率。
- 失败订单数。
- 按 `error_code` 聚合的失败分布。
- 按入口渠道、租户、应用版本、依赖服务聚合的失败分布。

如果无法直接提取 `error_code`，尝试从 Trace span、日志关键词和返回码推断错误族。仍无法推断时，升级 InvestigationAgent。

### 3. 路由 Runbook 分支

根据错误码分布进入分支。默认只处理库存锁定失败、优惠/价格校验失败两个经典分支。线上一次订单失败率告警通常只会出现 1-2 个主错误码；如果出现大量不同错误码，先判断是否为数据归类问题、错误码上报问题或更复杂的未知故障。

多个默认分支同时超过阈值且没有依赖顺序时，调用子 Agent 并行取证。

| 分支 | 错误码族 | 必查证据 |
|---|---|---|
| 库存锁定失败 | `INV_LOCK_FAIL`、`SKU_STOCK_NOT_ENOUGH`、`STOCK_VERSION_CONFLICT` | SKU TopN、库存锁定成功率、库存服务日志、库存版本冲突、`lockStock` Trace span、库存 DB 锁等待或缓存锁冲突 |
| 优惠/价格校验失败 | `PROMO_PRICE_MISMATCH`、`COUPON_INVALID`、`PROMO_RULE_NOT_MATCH` | 优惠规则 ID、价格快照版本、券状态、promotion-service Trace、规则发布时间、配置灰度范围 |

如果客户已经为支付/风控、幂等、数据库写入、配置缺失等错误码配置了扩展 Skill，可加载对应扩展 Skill；否则升级 InvestigationAgent。

### 4. 子 Agent 并行取证

子 Agent 只输出分支级结果，不直接给全局根因。

每个子 Agent 返回：

- 命中状态：主导、并发、排除、证据不足。
- 支撑证据：至少包含指标、日志、Trace、配置、业务状态中的两类。
- 反证：与该分支不一致的证据。
- 缺口：缺失数据源、无法确认的实体关系或时间窗。
- 分支结论：一句话说明该分支是否解释订单失败。

### 5. 主 Agent 汇总证据板

主 Agent 汇总所有分支，判断：

- 主导错误码和主导分支。
- 并发问题和噪声分支。
- 订单失败影响范围。
- 业务入口、渠道、租户、版本和依赖路径。
- 是否需要升级 InvestigationAgent。

每个根因候选至少需要两类独立证据。只有错误码占比、单一日志关键词或单一指标突增时，不输出确定性根因结论。

## 输出结构

按自然语言输出诊断报告：

```text
诊断结论：
<主因、并发问题、置信度，或升级 InvestigationAgent 的原因>

业务影响：
<订单创建失败率、失败订单数、影响入口、影响渠道、影响时间窗>

错误码分布：
<Top 错误码、占比、基线对比、命中的 Runbook 分支>

影响路径：
<OrderCreate -> API -> order-service -> 分支依赖 -> 证据来源>

分支证据板：
- 库存分支：<证据 / 反证 / 缺口 / 结论>
- 优惠价格分支：<证据 / 反证 / 缺口 / 结论>
- 其它错误码：<是否有扩展 Skill / 是否升级 InvestigationAgent>

建议动作：
<只读建议、需要人工确认的止血动作、后续治理动作>

升级判断：
<是否需要 InvestigationAgent 继续动态调查，以及原因>
```

## 升级 InvestigationAgent 条件

- 错误码缺失，且无法从 Trace、日志或返回码推断错误族。
- 错误码不在已登记 Runbook 分支内。
- 多个分支证据互相矛盾，无法判断主因和并发问题。
- 一跳订单链路无法解释异常，需要扩展到多跳依赖、DevOps 变更、业务活动或外部系统。
- 关键证据持续缺失，当前 Skill 无法形成可信结论。

## 边界

- 只读诊断，不执行生产变更。
- 修复建议必须标注为建议，由用户进入自己的变更流程确认。
- 涉及用户、订单、金额等敏感信息时，只展示脱敏标识和聚合统计。
