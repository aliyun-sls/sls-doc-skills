# 接入外部 MCP 工具 Guide Skill

> 本仓 **Guide Skill** 入口：把人读版实践文档（article，`https://sls.aliyun.com/doc/starops/practices/mcp-integration/article.html`）的 SOP 固化为 Agent 可加载形态，与 article 一一对应。
>
> 区分：本实践无 meta skill 样品。当前证据由 `verification.md` 结构化摘要和 `assets/thread-data/` 原始 replay data 支撑，不另起 `meta-skill-sample/`。
>
> 当前形态：覆盖 K8s MCP 与 GitLab MCP 验证摘要，并把内网可达性、受控动作和研发上下文三类能力边界沉淀为接入 SOP。未验证范围以 `design.md` 和 `verification.md` 为准。

## 何时触发本 Guide Skill

以下提问应路由到本 Guide Skill 的**主路径**：

- "帮我把外部 MCP 接入 STAROps"
- "把 K8s / GitLab / 内部系统 MCP 接进数字员工"
- "数字员工怎么调用外部工具"
- "怎么让数字员工查内部 CMDB / GitLab commit / Jenkins build"
- "MCP Server 怎么挂到 STAROps"
- "想把第三方 SaaS 通过 MCP 协议接进来"

以下提问应路由到本 流程的**治理段**（§治理）：

- "MCP 接入前要检查哪几项"
- "查询类 / 变更类 MCP 怎么选执行策略"
- "Bearer Token 过期了怎么续"
- "VPC 模式 vs 直连模式怎么选"

以下提问应路由到本 流程的**反向接入末段简介**：

- "STAROps 能不能当 MCP Server 被外部 Agent 调用"
- "Cursor / Claude Desktop 怎么接 STAROps"
- "`alibabacloud-starops-chat` 是什么"

以下提问不应路由到本 SOP：

- 接入后用 MCP 工具做告警 RCA → `alert-rca-flow`
- 服务方没有 MCP 计划，需要自定义 Skill 包装 → `skill-authoring` / `rds-inspection-via-script`
- 一次性 OpenAPI 查询，无复用价值 → chat 内直接提问
- 接入后想配长期任务调度 + 报告通知 → `log-insight-pattern`（场景卡路线）

## 主路径 SOP 概览

6 步配置流程，章节顺序与人读版实践文档（`https://sls.aliyun.com/doc/starops/practices/mcp-integration/article.html`）的部署与配置流程一一对应：

1. 确认能力归属：判断目标能力应走 STAROps 主诊断链、UModel 建模、MCP tool，还是 Guide/Runtime Skill。
2. 部署 MCP Server：确认网络位置、协议、认证、权限和日志。
3. 设计 tool 边界：写清用途、输入、输出、权限、风险、失败语义和验证方式。
4. 接入 STAROps：填写服务名称、访问端点、传输协议和认证方式，并执行工具发现。
5. 配置执行策略：按 tool 风险配置自动执行、谨慎执行、每次询问或关闭。
6. 挂载数字员工并验证：覆盖工具发现、正确选择、结果综合、失败降级和策略拦截。

## 步骤 1：打开「添加自定义 MCP」弹窗 + 选网络模式

**Agent 动作**：引导用户进入 STAROps 控制台左侧导航「数字员工」→ 选择目标数字员工 → 工具 tab → MCP 子 tab → 右上角「添加自定义 MCP」按钮。弹窗内先填两个名字字段 + 选网络模式。

**Agent 引导话术**：

> "请打开 STAROps 控制台，左侧导航点击「数字员工」，选择目标数字员工后进入「工具」tab，再切换到「MCP」子 tab，右上角「添加自定义 MCP」按钮会弹出配置弹窗。弹窗内先填两个标识字段：服务名称（英文唯一，如 `k8s_mcp` 或 `gitlab_mcp`）+ 显示名称（UI 展示用）+ 服务描述（可选，便于辨识）。然后选择网络模式：MCP Server 部署在公网走「直连模式」（只填访问端点 URL），部署在私网走「VPC 模式」（需填 VPC ID / 交换机 ID / 安全组 ID / 服务 IP + 端口 5 个字段）。"

**询问清单**：

| 询问项 | 用户回答示例 | 不指定时默认值 |
|---|---|---|
| 服务名称 | `k8s_mcp` / `gitlab_mcp` | 无默认，必填，英文唯一不可修改 |
| 显示名称 | `K8s MCP` / `GitLab MCP` | 默认同服务名称 |
| 服务描述 | `K8s operations MCP` / `GitLab context MCP` | 可选 |
| 网络模式 | 直连模式 / VPC 模式 | 无默认，必选；按 MCP Server 部署位置决定 |

**输出要求**：

- 服务名称 + 显示名称 + 网络模式 3 项明确
- 网络模式选定后，下方字段联动展开（VPC 模式露 5 个网络字段；直连模式只露访问端点 1 个字段）

**L0 声明**：本步骤只读，不发起任何变更（弹窗内仅填表，未保存）。

## 步骤 2：填服务端点 + 传输协议 + 认证

**Agent 动作**：根据步骤 1 选定的网络模式，引导用户填以下字段。直连模式 4 个字段，VPC 模式 4 个字段 + 步骤 1 的 5 个网络字段。

**询问清单（直连模式）**：

| 询问项 | 用户回答示例 | 注意事项 |
|---|---|---|
| 访问端点 | `https://dashscope.aliyuncs.com/api/v1/mcps/WebSearch/sse` | SSE 协议结尾是 `/sse`；HTTP 协议结尾是 `/mcp` |
| 传输协议 | `SSE` 或 `HTTP` | 必须与访问端点协议匹配 |
| 认证方式 | `Bearer Token` / `Custom Header` / `无` | 按 MCP Server 要求选择 |
| Token / Header 值 | 密码框，加密存储 | Bearer 类需确认 Token 有效期 + 续期机制 |

**Agent 引导话术**：

> "请提供 MCP Server 的访问端点 URL。SSE 协议的端点通常以 `/sse` 结尾，HTTP 协议以 `/mcp` 结尾；传输协议下拉需要与端点协议一致。认证方式按 MCP Server 要求选择，Bearer Token 是最常见模式，Token 填进密码框后服务端加密存储。"

**门控规则**（Agent 必须强制执行）：

- 端点 URL 协议匹配检查：访问端点以 `/sse` 结尾时传输协议必须选 `SSE`；以 `/mcp` 结尾时必须选 `HTTP`
- Token 字段提醒：Bearer Token 是密码框，Agent 不读取不回显；用户填完后只确认「已填」，不读 Token 内容

**输出要求**：

- 4 个字段填完，弹窗下方「拉取工具列表」按钮变为可点击状态

**L0 声明**：本步骤只读，不发起任何变更（弹窗内仅填表，未保存）。

## 步骤 3：点「拉取工具列表」自动发现工具

**Agent 动作**：引导用户点「拉取工具列表」按钮，平台向 MCP Server 发握手请求（MCP 协议 `tools/list`），返回该 Server 暴露的所有工具。Agent 协助勾选工具。

**Agent 引导话术**：

> "请点弹窗下方「拉取工具列表」按钮，平台会向 MCP Server 发握手请求并返回该 Server 暴露的所有工具。工具列表加载完成后，您可以全选，也可以按需要勾选部分工具。勾选意味着把该工具暴露给数字员工。请告诉我您想勾选哪些工具，或者全选。"

**失败处置**：

| 失败现象 | 排查顺序 |
|---|---|
| 按钮点了无反应 | 检查 4 个字段是否都填完；浏览器 console 看是否有网络错误 |
| 握手请求超时 | 直连模式：检查公网可达性；VPC 模式：检查安全组是否放行 STAROps 出口 IP |
| 返回错误：认证失败 | Token 过期 / Token 拼写错误 / 认证方式选错 |
| 返回错误：端点不可达 | 端点 URL 拼写检查（SSE 结尾 `/sse`，HTTP 结尾 `/mcp`）+ 传输协议下拉是否匹配 |
| 返回空列表 | MCP Server 当前未暴露任何工具，联系服务方确认 |

**输出要求**：

- 弹窗内出现「工具 (N/M)」标签（N=已勾选数，M=总数），N ≥ 1
- 勾选的工具会进入下一步执行策略配置

**L0 声明**：本步骤只读，平台对 MCP Server 的握手请求是只读查询（`tools/list`），不触发任何工具的实际调用。

## 步骤 4：给每个工具配执行策略

**Agent 动作**：对每个勾选的工具，按 §治理 中 MCP 4 分类决策表匹配执行策略。

**询问清单**：

| 询问项 | 用户回答示例 | Agent 推荐默认 |
|---|---|---|
| 每个工具的类型 | 查询类 / 分析类 / 变更类 / 高风险类 | Agent 按工具命名推断（`read_*` / `get_*` / `search_*` → 查询；`create_*` / `update_*` / `delete_*` → 变更）|
| 每个工具的执行策略 | 每次询问 / 自动执行 / 谨慎执行（默认）| 查询类 → 自动执行；分析类 → 自动或谨慎；变更类 → 每次询问；高风险类 → 强制每次询问 + 后续配 HIL |

**3 个策略选项**：

| 策略 | 行为 | 适用 MCP 类型 |
|---|---|---|
| 每次询问 | 数字员工调工具前在 chat 里弹卡片，用户点确认才执行 | 变更类（P1）/ 高风险（P2） |
| 自动执行 | 数字员工自主决定调用，不打断 | 查询类（P0）/ 分析类（P0） |
| 谨慎执行（默认） | 数字员工自主决定调用，但调用前在 chat 流式输出说明 | 不确定时的安全档 |

**Agent 引导话术**：

> "弹窗底部每个勾选的工具旁边都有「工具执行策略」下拉。我按工具命名推断了类型，给出推荐策略：`<工具名>` 推断为 `<类型>`，建议选「<策略>」。如果您觉得分类不对，可以追加说明（例如『这个 search_*  工具其实会修改索引』），我会重新匹配策略。"

**门控规则**（Agent 必须强制执行）：

- 变更类 / 高风险类工具的执行策略**不能**自动设为「自动执行」；如果用户明确要求，必须先在 chat 内提示风险 + 等用户二次确认
- 同一 MCP 下不同工具可配不同策略（如 `read_*` 走自动、`write_*` 走每次询问），不要求统一

**输出要求**：

- 所有勾选工具都设了策略，弹窗底部「保存」按钮可点击
- 点保存后回到 MCP 列表页，新接入的 MCP 显示在列表里且状态正常（无 error badge）

**L0 声明**：保存是创建配置操作（不动客户业务数据），属 L0 风险范围内。

## 步骤 5：挂数字员工 + chat 验证

**Agent 动作**：MCP 在「添加自定义 MCP」入口下接入时默认挂到当前数字员工；如果是从 MCP 列表页统一管理后接入，需手动在数字员工配置页勾选 MCP。然后引导用户在 chat 内提问触发工具调用。

**Agent 引导话术**：

> "接入完成后我们在 chat 里验证一次工具调用。请告诉我您想测试的工具用途，我帮您组织一句触发 prompt（例如查询类工具：『请使用 <MCP 服务名> 在 <来源> 搜索 <关键词>，综合返回结果给我说明...』）。提问后请确认两点：（1）chat 内是否出现工具调用卡片（含输入参数 + 输出结果摘要）；（2）数字员工是否把工具结果综合进了回答。两者都到 → 接入成功。"

**测试 prompt 模板**（按工具类型）：

| 工具类型 | prompt 模板 |
|---|---|
| 查询类 | `请使用 <MCP 服务名> 在 <来源> 搜索/查询 <关键词>，综合返回结果说明 <场景>` |
| 分析类 | `请使用 <MCP 服务名> 对 <对象> 做 <分析维度>，给出 <结论形式>` |
| 变更类 | `请使用 <MCP 服务名> 创建 / 更新 <对象>（参数：<参数列表>），先告诉我预期影响范围再执行` |

**输出要求**（首次 chat 验证）：

- 工具调用卡片可见（含输入参数 + 输出结果摘要）
- 工具结果被综合进数字员工的回答（不是直接堆放原始返回值）
- 执行策略生效（自动执行下不打断；每次询问下弹卡片）

**L0 声明**：查询类 / 分析类工具调用本身是只读；变更类 / 高风险类工具调用需用户在「每次询问」卡片上点确认才会真正执行。

## 治理：5 项接入前检查 + MCP 4 分类决策

每次接入新 MCP 之前必须过 5 项检查（事实源 = `strategy-2026q2.md` §MCP 建设）：

| 检查项 | 要求 |
|---|---|
| 网络 | 明确公网 / VPC / AI 网关 / 函数计算接入方式 |
| 权限 | 工具级最小权限，避免一个 MCP 暴露过大能力（Token 范围、命名空间隔离） |
| 审计 | 所有调用能追踪到用户 / 数字员工 / 任务 / 参数（平台默认审计 + 后台执行日志） |
| 幂等 | 写操作需说明是否可重试、如何回滚（查询类不适用） |
| 降级 | 工具不可用时不阻塞整体对话（数字员工自主降级到内置知识或其他工具） |

MCP 类型 4 分类决定执行策略选择：

| MCP 类型 | 优先级 | 示例 | 默认执行策略 | 接入门槛 |
|---|---|---|---|---|
| 查询类 | P0 | 查询 CMDB / GitLab commit / Jenkins build / 工单状态 | 自动执行 | 接入即用 |
| 分析类 | P0 | 代码搜索 / 日志样本提取 / 配置差异分析 | 自动执行 或 谨慎执行 | 接入即用 |
| 变更类 | P1 | 创建工单 / 暂停流水线 / 扩缩容 | 每次询问 | 接入前评审 + 审批闭环 |
| 高风险 | P2 | 删除资源 / 强制回滚 / 重启核心服务 | 强制 HIL + 二次确认 | 单独评审 + 回滚演练 |

## 反向接入末段简介

STAROps 也可以**被**接入到外部智能体作为 MCP Server。外部智能体（Cursor / Claude Desktop / 自研 Agent）可以通过 MCP 协议调用 STAROps 提供的能力（指标查询 / Skill 触发 / 数字员工对话）。

这条路线的官方包名为 `alibabacloud-starops-chat`，提供 chat 转发能力让外部智能体把 STAROps 当成一个 chat-style MCP Server 来用。

Agent 引导话术：

> "您问的是反向接入：STAROps 作为 MCP Server 被外部智能体调用，方向与本 SOP 相反。这条路线的官方包名为 `alibabacloud-starops-chat`，本 SOP 不展开反向接入流程。如有需求，请参考 STAROps 控制台 `外部 AI Agent 集成` 页面 + 阿里云 STAROps 官方集成文档；如果反向接入是您的核心需求，建议拆独立工单沟通。"

## 输出与交付

主路径 SOP 跑完后客户能拿到：

- **新接入的 MCP**：在 MCP 列表页可见，状态正常，可被数字员工调用
- **工具级执行策略**：每个勾选工具的策略明确（每次询问 / 自动执行 / 谨慎执行）
- **chat 内可调用**：挂载的数字员工 chat 内提问可触发工具调用，结果综合进对话回答
- **接入前 5 项检查证据**：网络 / 权限 / 审计 / 幂等 / 降级 5 项核对结论（口头确认或留存到内部审批工单）

## 失败与回滚

L0 只读 + 配置创建，无回滚需求（删除 MCP 配置回到接入前状态即可）。常见失败处置：

| 步骤 | 可能失败 | 处置 |
|---|---|---|
| 步骤 1 | 弹窗加载不出 | 检查 STAROps 实例是否开通「自定义 MCP」管理权限；联系平台团队 |
| 步骤 1 | 网络模式选错 | MCP 列表页打开编辑入口（与新建共用同一弹窗）→ 重选网络模式 → 保存 |
| 步骤 2 | 端点协议不匹配 | 检查 SSE 端点结尾是否 `/sse`、HTTP 端点结尾是否 `/mcp`；传输协议下拉对应调整 |
| 步骤 2 | Token 过期 | MCP 编辑页重填 Token → 保存；生产场景配 Token 续期提醒 |
| 步骤 3 | 拉取工具列表超时 | VPC 模式：检查安全组放行 STAROps 出口 IP；直连模式：检查公网可达性 |
| 步骤 3 | 工具列表为空 | MCP Server 未暴露工具，联系服务方确认 |
| 步骤 4 | 变更类工具误设「自动执行」 | MCP 编辑页改回「每次询问」+ 在审批工单登记策略变更 |
| 步骤 5 | chat 内数字员工不调用工具 | 检查：MCP 是否挂到当前数字员工 → 工具描述是否清晰 → prompt 是否给了触发提示（如 `请使用 <MCP 服务名>`）|
| 任意 | 接入后想完全撤销 | MCP 列表页删除该 MCP；删除是 L0 操作（仅删除配置，不影响 MCP Server 自身）|

控制台 chat 内 @数字员工：「我刚配的 `<MCP 服务名>` 拉不到工具」/「调用失败」/「Token 要过期了怎么续」→ 数字员工持有当前 MCP 配置上下文，会直接定位问题或查后台错误日志。
